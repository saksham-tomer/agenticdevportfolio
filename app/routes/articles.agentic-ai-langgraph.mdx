## Why LangGraph Changes Everything

Building a production AI agent used to mean spaghetti code: chains of `if` statements, brittle prompt concatenation, and no real error handling. LangGraph solves this by modeling agent behavior as a **directed graph** where nodes are functions, edges are routing decisions, and state flows through explicitly.

I used LangGraph to build the AI game engine at OpenBook.games — a multi-agent system that runs at 99.5% availability on AWS EKS. This is how it works.

## What Is LangGraph?

LangGraph is a library from LangChain that lets you build **stateful, multi-step LLM workflows** as graphs. Unlike linear chains, agents can:

- Loop back to retry failed steps
- Branch based on LLM output
- Run multiple subgraphs in parallel
- Pause and resume (for human-in-the-loop)

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict

class AgentState(TypedDict):
    messages: list
    context: str
    action_taken: str
    iteration: int
```

The `AgentState` is passed between every node. Every node reads from it and returns a partial dict that updates it.

## Building a Minimal ReAct Agent

The Reasoning + Acting (ReAct) pattern is the most common agent loop:

```python
from langgraph.graph import StateGraph, END
from langchain_core.messages import HumanMessage, AIMessage
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def agent_node(state: AgentState) -> dict:
    """Core reasoning step."""
    response = llm.invoke(state["messages"])
    return {"messages": [*state["messages"], response]}

def tool_node(state: AgentState) -> dict:
    """Execute tool calls from the agent's last message."""
    last_message = state["messages"][-1]
    # Execute tool calls, append results
    results = execute_tools(last_message.tool_calls)
    return {"messages": [*state["messages"], *results]}

def should_continue(state: AgentState) -> str:
    """Route: continue looping or exit."""
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return "end"

# Build the graph
workflow = StateGraph(AgentState)
workflow.add_node("agent", agent_node)
workflow.add_node("tools", tool_node)

workflow.set_entry_point("agent")
workflow.add_conditional_edges("agent", should_continue, {
    "tools": "tools",
    "end": END,
})
workflow.add_edge("tools", "agent")  # loop back

app = workflow.compile()
```

This compiles to an executable with proper cycle detection and state management.

## Multi-Agent Architecture

For the game engine, I needed multiple specialized agents communicating via a shared memory bus:

```
┌─────────────────────────────────────┐
│            Supervisor Agent          │
│   (routes tasks to sub-agents)      │
└──────┬──────────┬────────┬──────────┘
       │          │        │
  ┌────▼──┐  ┌───▼──┐  ┌──▼─────┐
  │ NPC   │  │ Scene│  │ Audio  │
  │ Agent │  │ Agent│  │ Agent  │
  └───────┘  └──────┘  └────────┘
```

```python
from langgraph.graph import StateGraph
from langgraph.prebuilt import ToolNode

# Each subgraph is its own compiled workflow
npc_subgraph = build_npc_agent()
scene_subgraph = build_scene_agent()

def supervisor(state):
    task = state["task"]
    if "npc" in task:
        return {"route": "npc"}
    elif "scene" in task:
        return {"route": "scene"}
    return {"route": "audio"}

main_graph = StateGraph(AgentState)
main_graph.add_node("supervisor", supervisor)
main_graph.add_node("npc_agent", npc_subgraph)
main_graph.add_node("scene_agent", scene_subgraph)

main_graph.add_conditional_edges("supervisor", lambda s: s["route"], {
    "npc": "npc_agent",
    "scene": "scene_agent",
})
```

## Persistence with Checkpointing

For production, agents need to survive restarts. LangGraph uses **checkpointers** backed by a database:

```python
from langgraph.checkpoint.postgres import PostgresSaver

pool = ConnectionPool("postgresql://...")
checkpointer = PostgresSaver(pool)

app = workflow.compile(checkpointer=checkpointer)

# Resume a conversation by thread_id
config = {"configurable": {"thread_id": "user-session-42"}}
result = await app.ainvoke({"messages": [HumanMessage("Continue")]}, config)
```

Every step is automatically saved. If the container crashes, the agent resumes from its last checkpoint.

## Error Handling and Retries

Production agents must handle LLM failures gracefully:

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(min=1, max=10))
async def resilient_llm_call(messages):
    return await llm.ainvoke(messages)

def agent_node_with_retry(state):
    try:
        response = await resilient_llm_call(state["messages"])
        return {"messages": [...state["messages"], response], "error": None}
    except Exception as e:
        return {"error": str(e), "iteration": state["iteration"] + 1}
```

Add a guard node that routes to a fallback if `error` is set after 3 iterations.

## Deploying on AWS EKS

The game engine runs as a set of microservices, each wrapping a compiled LangGraph agent:

```yaml
# agent-deployment.yaml
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: npc-agent
        image: openbook/npc-agent:latest
        env:
        - name: POSTGRES_URL
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: url
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
```

We use HPA (Horizontal Pod Autoscaler) tied to a custom metric (agent queue depth) to scale from 3 to 30 replicas under load.
